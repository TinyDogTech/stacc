# README + distribution refresh

## Summary

Refresh stacc's README into an accurate, dual-audience front door, written for
both humans and AI agents, in the tone Astral uses for ruff/uv, with a guided
workflow walkthrough modeled on Graphite's CLI quick-start. Alongside it, add a
real distribution path (prebuilt binaries, a curl install script, and Homebrew,
generated by cargo-dist) so the install story the README documents is true.

Two related deliverables. Distribution is sequenced first (or in lockstep) so the
README never documents an install path that does not yet exist.

## Problem

- **The README is stale.** It still describes an early milestone, omits most of
  the command surface (the daily edit loop, navigation, lifecycle, recovery), and
  barely surfaces the agent-first JSON/non-interactive design that is the point of
  the tool.
- **There is no install story.** The only documented path is `cargo build` from
  source. There is no clean "install stacc," and an agent cannot grab a working
  binary without a Rust toolchain and a compile.
- **stacc serves two audiences and the README speaks to neither well.** Humans
  adopt and set it up; agents operate it (non-interactive, `--format json`,
  `--help`). The README should make both first-class.

## Goals

- A README that is accurate, benefit-led, and explicitly useful to both humans
  and agents.
- A guided workflow walkthrough that carries a reader from nothing to a merged
  stack, using non-interactive invocations and showing a sample `--format json`
  output so it doubles as agent material.
- An agent-driving block a human can drop into their agent's context: the JSON
  contract, the structured-error shape, and the conflict recovery protocol.
- A real install story: a one-line curl installer and `brew install`, backed by
  prebuilt release binaries, with a from-source fallback.

## Non-goals (scope boundaries)

- **No `cargo install` / crates.io** for now. It would mean publishing all six
  workspace crates and still compiles on the user's machine, the opposite of the
  simple, no-toolchain path agents need. Deferred, not refused.
- **No competitor comparisons.** The README sells stacc on its own merits and
  does not name other tools.
- **No version numbers or milestone labels in the status section.** Convey
  maturity in prose, not "v1" / milestone identifiers.
- **No exhaustive per-command flag reference in the README.** The walkthrough
  covers the workflow; `--help` remains the source of truth for flags and args.
- **Not building a "stacc describes itself" capability** (e.g. a command that
  prints an agent-ready manual). Noted as a possible future enhancement only.

## Requirements

### README (R1-R10)

- **R1. Hero.** Name, an agent-first one-line description, and a plain statement
  that stacc is for both humans and AI agents.
- **R2. Why stacc.** Sell on its own merits: non-interactive by default,
  machine-readable JSON output, structured errors, branch-per-PR. No competitor
  framing.
- **R3. Install.** The curl one-liner as the primary path, `brew install`, and a
  from-source fallback for contributors.
- **R4. Workflow walkthrough.** Graphite-quick-start-style annotated terminal
  blocks carrying the full lifecycle end to end: initialize, create a branch,
  modify, submit, stack a second PR, sync, merge, plus navigation and recovery.
  Uses non-interactive invocations and includes at least one `--format json`
  output sample. Defers deep flags to `--help`.
- **R5. Driving stacc from an agent.** A dedicated, copy-pasteable section: the
  JSON output contract (`--format json` on every command), the structured-error
  shape, and the `continue`/`abort` recovery protocol. Framed as something a
  human pastes into their agent's context / AGENTS.md.
- **R6. Authentication.** Keep, but trim the existing section to the essentials.
- **R7. How it works (brief).** The `refs/stacc/` state ref and the
  branch-per-PR model, enough for a reader to trust it, not a design doc.
- **R8. Development.** Build/test/clippy and the workspace crate table, moved
  into a contributor-facing section rather than sitting near the top.
- **R9. Status.** Replace the stale milestone text with an accurate,
  version-free statement of maturity, plus a link to the design doc for depth.
- **R10. Tone.** Match Astral's docs voice: confident, concise, benefit-led,
  direct second person, no hyperbole or filler.

### Distribution (R11-R14)

- **R11. Release binaries.** cargo-dist config that builds cross-platform
  prebuilt binaries and creates a GitHub Release on each tag.
- **R12. Install script.** The cargo-dist-generated `curl | sh` (and PowerShell)
  installer that downloads the right prebuilt binary; this is the README's hero
  install line.
- **R13. Homebrew.** A cargo-dist-generated Homebrew formula so `brew install`
  works (near-zero marginal cost once cargo-dist is configured).
- **R14. Sequencing.** Distribution lands before, or together with, the README so
  the documented install path is real.

## Success criteria

- A new human can install stacc with one curl command (or `brew install`) and
  follow the walkthrough to submit and merge a stack.
- An agent pointed at the README's agent block plus `--help` can drive stacc
  non-interactively and parse its JSON output.
- The README contains no stale status, no version numbers, and no competitor
  names, and reads in Astral's tone.
- `curl ... | sh` installs a working binary on macOS and Linux (and the
  PowerShell variant on Windows); `brew install` installs the same binary.

## Open questions and assumptions

- **Install URL.** Assume the cargo-dist default (GitHub Releases download URL);
  no custom domain. A short hosted URL is a possible later nicety.
- **Agent block placement.** Assume it lives in the README as a copy-pasteable
  section. A self-describing command remains a future enhancement.
- **Homebrew tap.** Assume a tap repo that cargo-dist manages; confirm the tap
  name during planning.
- **Status wording.** Verify the exact maturity framing against the repo state
  when writing (all core commands are implemented).

## Dependencies

- cargo-dist (actively maintained) and a tag-based GitHub Actions release
  workflow.
- A Homebrew tap repository for the generated formula.
- The README's install section depends on R11-R13 being in place (R14).
